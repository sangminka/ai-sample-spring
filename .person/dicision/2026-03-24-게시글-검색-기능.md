# 게시글 검색 기능

- 날짜: 2026-03-24
- 참여: 사용자 + Codex

## 배경

게시글 목록 기능에 검색 기능을 단계적으로 추가하기로 했다.
한 번에 복잡하게 구현하지 않고, 학습이 쉬운 순서대로 `SSR 검색 -> AJAX 실시간 검색 -> 디바운스 적용` 흐름으로 확장하는 방향에 합의했다.
현재 프로젝트 규칙상 기본 요청은 `form` 기반 SSR이 자연스럽고, AJAX는 필요한 시점에만 `ApiController`를 사용하는 방식이 프로젝트 규칙과도 잘 맞는다.

## 핵심 논의

### 1. 1단계는 SSR 검색으로 시작한다

- 요청 방식은 `GET /board/list?page=1&keyword=...`
- 목록 페이지에 검색창 1개와 검색 버튼을 둔다
- `<form method="get" action="/board/list">` 구조로 처리한다
- 검색창은 `제목 + 내용` 통합 검색으로 시작한다

### 2. 검색 규칙

- 검색어는 앞뒤 공백을 `trim` 한다
- `trim` 후 빈 문자열이면 검색이 아니라 전체 목록을 보여준다
- 검색 조건은 `title like %keyword% or content like %keyword%`
- 정렬은 기존 목록과 동일하게 `id desc` 최신순을 유지한다

### 3. 상태 유지 규칙

- 검색 후 페이지 이동 시 `page + keyword`를 함께 유지한다
- 상세 페이지 진입 후 목록 복귀 시에도 `page + keyword`를 함께 유지한다
- 검색어가 바뀌면 항상 `page=1`로 다시 시작한다

### 4. 검색 결과 없음 처리

- 검색 결과가 없으면 검색어를 포함한 문구를 보여준다
- 예시: `'스프링' 검색 결과가 없습니다`

### 5. 2단계 AJAX 방향

- 이벤트는 `input`을 사용한다
- 요청은 `GET /api/board/search?page=1&keyword=...`
- 응답은 `Resp.ok(BoardResponse.ListResponse)` 형태로 한다
- SSR과 AJAX가 같은 응답 구조를 재사용한다
- 검색 입력만 AJAX로 처리하고, 검색 버튼 제출과 페이지 이동은 당분간 SSR을 유지한다
- AJAX 성공 시 `목록 + 빈 결과 문구 + 페이지네이션`을 함께 다시 그린다

### 6. 3단계 디바운스 방향

- 디바운스 시간은 `300ms`
- 입력이 비면 전체 목록으로 복귀한다
- 로딩 표시는 디바운스가 끝나고 실제 요청이 시작될 때만 보여준다
- 실패 시 기존 목록은 유지하고, 검색창 근처에 오류 메시지만 보여준다
- 마지막 요청 결과만 화면에 반영한다
- URL은 `history.replaceState`로 `?page=1&keyword=...` 형태를 유지한다

## 결론

구현은 다음 순서로 진행한다.

1. 1단계 SSR 검색을 먼저 완성한다
2. 1단계에서 만든 검색 DTO, 서비스 규칙, 응답 DTO를 재사용하여 2단계 AJAX를 붙인다
3. 마지막으로 3단계에서 `300ms` 디바운스와 마지막 요청 우선 처리까지 적용한다

핵심 원칙은 `검색 조건 구조는 유지하고, 호출 방식만 단계적으로 발전시키는 것`이다.
