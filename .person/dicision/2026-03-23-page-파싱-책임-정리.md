# page 파싱 책임 정리

- 날짜: 2026-03-23
- 참여: 사용자 + Codex

## 배경

`BoardController`에서 `page` 관련 메서드가 여러 개 존재했다.

- `페이지가비어있음(String page)`
- `페이지번호파싱하기(String page)`
- `상세페이지번호보정하기(String page)`

이 중 어떤 로직이 컨트롤러에 남아야 하고, 어떤 로직은 Service와 중복되는지 정리할 필요가 있었다.

특히 상세 페이지의 `page`는 목록 조회용 핵심 입력인지,
아니면 목록으로 돌아가기 위한 문맥 정보인지 역할을 다시 구분할 필요가 있었다.

## 핵심 논의

- `list`의 `page`는 실제 조회 결과를 바꾸므로 엄격하게 검증해야 한다.
- `detail`의 `page`는 게시글 조회의 본질이 아니라 목록 복귀용 문맥 정보다.
- 목록에서 상세로 이동하는 정상 흐름에서는 이미 보정된 `page`가 전달되므로,
  상세 쪽에서 별도 보정 메서드를 두는 것은 과할 수 있다.
- `String -> Integer` 파싱은 웹 요청 처리 공통 로직이므로 컨트롤러 공용 메서드로 둘 수 있다.
- 최종 페이지 보정 규칙(`null`, `1 미만 -> 1`)은 Service의 `페이지보정하기(Integer page)` 하나로 통일하는 편이 낫다.

## 결론

- `페이지번호파싱하기(String page)`는 컨트롤러 공용 메서드로 유지한다.
- `list`는 `page`를 엄격하게 검증하고, 잘못된 값이면 `/board/list?page=1`로 `redirect`한다.
- `detail`은 `page`를 단순 파싱만 해서 Service로 넘긴다.
- `detail`의 최종 `page` 보정은 Service의 `페이지보정하기(Integer page)`가 담당한다.
- `상세페이지번호보정하기(String page)`는 제거해도 된다.

## 권장 구조

```java
@GetMapping("/board/list")
public String list(@RequestParam(required = false) String page, Model model) {
    if (page == null || page.isBlank()) {
        return "redirect:/board/list?page=1";
    }

    var pageNumber = 페이지번호파싱하기(page);
    if (pageNumber == null || pageNumber < 1) {
        return "redirect:/board/list?page=1";
    }

    model.addAttribute("model", boardService.게시글목록보기(pageNumber));
    return "list";
}

@GetMapping("/board/{id}")
public String detail(@PathVariable("id") Integer id,
        @RequestParam(required = false) String page,
        Model model) {
    model.addAttribute("model", boardService.게시글상세보기(id, 페이지번호파싱하기(page)));
    return "detail";
}

private Integer 페이지번호파싱하기(String page) {
    if (page == null || page.isBlank()) {
        return null;
    }

    try {
        return Integer.parseInt(page);
    } catch (NumberFormatException e) {
        return null;
    }
}
```

- `detail`은 컨트롤러에서 파싱만 하고, 최종 보정은 Service로 넘기는 방향으로 정리하면 된다.
