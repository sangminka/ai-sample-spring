# 페이징 정책

- 날짜: 2026-03-23
- 참여: 사용자 + Codex

## 배경

게시글 목록 페이징을 구현하는 과정에서 `page` 값을 `0`부터 시작할지, `1`부터 시작할지에 대한 기준을 정리할 필요가 있었다.
특히 Step 3의 `이전/다음 버튼`, Step 4의 `마지막 페이지 계산`, 잘못된 주소 처리 정책까지 함께 맞춰야 했다.

## 핵심 논의

현재 구현은 `0-based`에 가깝다.

- 기본값이 `page=0`
- 음수 페이지는 내부적으로 `0`으로 보정
- `offset = page * PAGE_SIZE`

하지만 사용자 관점에서는 `?page=1`이 더 자연스럽기 때문에, 외부에 보이는 페이지 번호를 `1-based`로 바꾸는 방향을 논의했다.

또한 잘못된 주소 처리에 대해서도 다음 두 종류를 구분했다.

- 형식이 잘못된 경우
  - 예: `page` 누락, `page=0`, `page=-1`, `page=abc`
- 형식은 맞지만 범위를 벗어난 경우
  - 예: `page=9999`

이 둘을 같은 방식으로 처리하면 불필요한 복잡성이 생기므로, 정책을 분리하는 것이 더 낫다는 결론에 도달했다.

## 결론

최종 합의한 페이징 정책은 다음과 같다.

- 페이지 번호는 URL, 화면, 서비스 모두 `1-based`로 사용한다.
- 첫 페이지의 정식 주소는 `/board/list?page=1`로 고정한다.
- `page < 1` 이거나 숫자 변환이 불가능한 형식 오류는 `/board/list?page=1`로 `redirect`한다.
- `page > 마지막 페이지`인 경우에는 마지막 페이지로 강제 이동시키지 않는다.
- 마지막 페이지를 초과한 경우는 빈 목록 또는 안내 문구를 렌더링한다.
- 상세 페이지에서 목록으로 돌아갈 때도 현재 `page` 값을 그대로 유지한다.

## 계층별 책임

컨트롤러의 책임:

- 잘못된 URL 파라미터를 판별한다.
- 필요 시 `/board/list?page=1`로 `redirect`한다.

서비스의 책임:

- 정상적인 `page` 값을 받아 조회용 `offset`을 계산한다.
- 계산식은 `offset = (page - 1) * PAGE_SIZE`를 사용한다.

원칙:

- `page`는 페이지 번호다.
- `offset`은 DB 조회용 파생값이다.
- 둘을 같은 의미로 섞어 쓰지 않는다.

## 단계별 적용 범위

Step 3에서 할 일:

- `1-based` 페이지 기준으로 전환
- 이전/다음 버튼 구현
- `page < 1` 및 형식 오류에 대한 `redirect` 처리

Step 4에서 할 일:

- `COUNT(*)`로 전체 개수 조회
- 마지막 페이지 계산
- `page > 마지막 페이지`일 때 빈 목록 또는 안내 문구 처리
