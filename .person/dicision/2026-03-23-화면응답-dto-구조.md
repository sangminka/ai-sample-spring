# 화면응답 DTO 구조

- 날짜: 2026-03-23
- 참여: 사용자 + Codex

## 배경

`BoardController`의 `list` 메서드에서 화면에 필요한 값을 여러 개 `model.addAttribute(...)`로 나누어 담기보다,
Service가 화면에 필요한 데이터를 하나의 DTO로 완성해서 넘기는 구조가 더 적절한지 논의했다.

또한 상세 화면에서 사용하는 `page` 값이 게시글 상세 데이터 자체인지,
아니면 목록으로 돌아가기 위한 화면 문맥 정보인지도 함께 구분할 필요가 있었다.

## 핵심 논의

- Controller는 가능한 한 얇게 유지하고 싶다.
- 다만 `redirect` 같은 HTTP 책임까지 Service로 넘기면 Service가 웹 계층에 과하게 결합된다.
- 따라서 Controller는 요청 파라미터 검증과 `redirect` 여부만 판단하고,
  Service는 정상 입력을 기준으로 화면에 필요한 데이터를 DTO 하나로 완성하는 구조가 적절하다고 보았다.
- 목록 화면의 페이징 정보는 공통 메타 정보로 분리할 수 있다.
- 상세 화면의 `page`는 게시글 데이터가 아니라 "목록으로 돌아가기 위한 문맥 정보"에 가깝다.
- 그래서 상세 화면에는 목록용 `PagingResponse` 전체를 넣지 않고, 필요한 `page` 정도만 둔다.

## 결론

최종 합의한 구조는 다음과 같다.

- Service는 화면에 필요한 데이터를 하나의 Response DTO에 담아 반환한다.
- Controller는 요청값 검증, `redirect` 처리, `model.addAttribute("model", dto)`만 담당한다.
- 목록 화면은 `ListResponse`와 `PagingResponse`를 조합해 표현한다.
- 상세 화면은 `DetailResponse`와 `page`를 사용하며, 목록용 `PagingResponse`는 넣지 않는다.
- 게시글 자체 데이터와 화면 이동 문맥 데이터는 의미를 구분해서 설계한다.

## 예시 구조

```java
public class BoardResponse {

    @Data
    public static class PagingResponse {
        private Integer currentPage;
        private Integer pageSize;
        private Integer totalCount;
        private Integer totalPageCount;
        private boolean hasPrev;
        private Integer prevPage;
        private boolean hasNext;
        private Integer nextPage;
    }

    @Data
    public static class ListResponse {
        private List<Min> boards;
        private PagingResponse paging;
    }

    @Data
    public static class DetailResponse {
        private Detail board;
        private Integer page;
    }
}
```
